### Step 1.3: Update `HRObserverRoom.jsx` imports

Same as Step 1.2 — replace `turnUtils` imports with `webrtcConfig` async variants.

### Step 1.4: Verify `webrtcConfig.js` is the single source of truth

`webrtcConfig.js` already:
- Fetches TURN credentials from the backend (`/api/turn-credentials`)
- Falls back to STUN-only if backend is unavailable
- Has caching, cooldown, and concurrent-fetch deduplication

No changes needed to this file — just confirm it's used everywhere.

### Step 1.5: Audit for any other references to `turnUtils`

Search the entire codebase for any remaining imports or references to `turnUtils` and remove them.

## Phase 2: Remove Hardcoded HR Credentials (Bug #3)

**Goal:** Eliminate the backdoor HR login that bypasses Firebase Auth entirely.

### Step 2.1: Remove hardcoded HR credentials from `AuthContext.jsx`

Remove:

Remove the localStorage-based mock HR logic from:

- `logout()` function (remove `localStorage.removeItem(STORAGE_KEY)`)
- `onAuthStateChanged` effect (remove mock HR restoration logic)

### Step 2.2: Create HR users via CLI script

Add a CLI script `server/scripts/create-hr-user.js` that:
1. Takes email + password as CLI args (`node create-hr-user.js hr@example.com MyP@ss123`)
2. Uses Firebase Admin SDK to create the user in Firebase Auth

The script should:
- Load `dotenv` and `firebase-admin` (already server dependencies)
- Reuse the same `FIREBASE_SERVICE_ACCOUNT_PATH` env var for initialization

- Case 1: Real Firebase user → `getIdToken()`

### Step 3.2: Fix `ProtectedRoute.jsx` — handle null role correctly (Bug #6)

Current logic allows access when `userRole` is null:
```js
if (requiredRole && userRole && userRole !== requiredRole) {
  return <Navigate to={...} replace />
}
```

Fixed — show loading state when role is unknown:
```jsx
export default function ProtectedRoute({ children, requiredRole }) {
  const { currentUser, userRole, loading } = useAuth()

  if (!currentUser) return <Navigate to="/login" replace />
  if (loading || userRole === null) {
    return <div style={{...spinner...}} /> // loading spinner
  }
  if (requiredRole && userRole !== requiredRole) {
    return <Navigate to={userRole === 'hr' ? '/hr' : '/candidate'} replace />
  }
  return children
}
```

This ensures that when `userRole` is null (e.g., Firestore timeout), the user sees a spinner instead of being granted access to protected routes.

### Step 5.3: Persist room state in Redis (Bug #11)

Add Redis-backed persistence for room state so interviews survive server restarts.

**5.3.1: Add `ioredis` dependency**

```bash
cd server && npm install ioredis
```

Add `REDIS_URL` to `.env.example` (default: `redis://localhost:6379`).

**5.3.2: Create `server/roomStore.js` — abstraction layer over in-memory + Redis**

```js
// roomStore.js — provides getRoom/getOrCreateRoom/setRoom/deleteRoom
// Uses Redis as primary store, in-memory as hot cache

const Redis = require('ioredis')
const redis = new Redis(process.env.REDIS_URL || 'redis://localhost:6379')
const ROOM_PREFIX = 'room:'
const ALIAS_PREFIX = 'alias:'

// In-memory cache for hot-path reads
const memoryCache = new Map()

async function getRoom(roomId) {
  // Check memory cache first
  if (memoryCache.has(roomId)) return memoryCache.get(roomId)
  // Fall back to Redis
  const raw = await redis.get(ROOM_PREFIX + roomId)
  if (!raw) return null
  const room = JSON.parse(raw)
  room.observers = new Set(room.observers || [])
  memoryCache.set(roomId, room)
  return room
}

async function setRoom(roomId, roomData) {
  const serializable = {
    candidate: roomData.candidate,
    hr: roomData.hr,
    observers: [...roomData.observers],
    candidateLocked: roomData.candidateLocked,
  }
  await redis.set(ROOM_PREFIX + roomId, JSON.stringify(serializable), 'EX', 86400) // 24h TTL
  memoryCache.set(roomId, { ...roomData })
}

async function deleteRoom(roomId) {
  await redis.del(ROOM_PREFIX + roomId)
  memoryCache.delete(roomId)
}

// Alias management
async function getAlias(aliasRoomId) {
  return await redis.get(ALIAS_PREFIX + aliasRoomId)
}

async function setAlias(aliasRoomId, sourceRoomId) {
  await redis.set(ALIAS_PREFIX + aliasRoomId, sourceRoomId, 'EX', 86400)
}

async function deleteAlias(aliasRoomId) {
  await redis.del(ALIAS_PREFIX + aliasRoomId)
}
```

**5.3.3: Update `server/server.js` to use `roomStore`**

- Replace direct `rooms[roomId]` reads with `await getRoom(roomId)`
- Replace `rooms[roomId] = ...` with `await setRoom(roomId, room)`
- Replace `delete rooms[roomId]` with `await deleteRoom(roomId)`
- Replace `observerAliases.get/set/delete` with `await getAlias/setAlias/deleteAlias`
- Keep `roomSockets` (socket tracking) in-memory since it's connection-scoped and doesn't need persistence

**5.3.4: Handle reconnection after server restart**

When a client reconnects after server restart:
- Socket reconnects automatically (socket.io reconnection)
- Client re-emits `join-room`, which calls `getOrCreateRoom`
- `getOrCreateRoom` finds the room in Redis, restores it
- `roomSockets` starts fresh (empty) — this is fine since old socket IDs are stale
- Client may need to re-send their stream (`request-stream` handles this)

**5.3.5: Update `render.yaml` for Redis**

Add Redis service to the Render deployment config:
```yaml
  - type: redis
    name: ai-interviewer-redis
    plan: free
    ipAllowList: []
```

Set `REDIS_URL` env var on the web service to reference the Redis instance.

**5.3.6: Graceful fallback if Redis is unavailable**

In `roomStore.js`, if Redis connection fails:
- Log a warning
- Fall back to in-memory only (current behavior)
- This ensures the app still works in development without Redis

## Risk Assessment

- **Phase 1-2 (Security):** High impact, low risk. Removing secrets is straightforward. The mock HR removal requires ensuring real HR users exist in Firebase before deploying.
- **Phase 3 (Auth):** Low risk. Simple filter additions and conditional logic.
- **Phase 4 (InterviewRoom):** Medium risk. Complex file with many interdependent effects. Changes should be tested thoroughly. The signaling disambiguation (Step 4.3) is the most delicate change.
- **Phase 5 (Server + Redis):** Medium risk. Adding auth to TURN endpoint requires frontend changes too. Rate limit cleanup is safe. Redis integration is additive — with graceful fallback, the worst case is reverting to in-memory behavior.
- **Phase 6 (Minor):** Very low risk. Cosmetic fixes.

---

## Execution Order

Execute phases sequentially: **1 → 2 → 3 → 4 → 5 → 6 → 7**

Each phase should be tested before moving to the next. Phase 1 and 2 are the most critical and should be deployed together to prevent any window where the old mock HR login still works but the API key is removed.
