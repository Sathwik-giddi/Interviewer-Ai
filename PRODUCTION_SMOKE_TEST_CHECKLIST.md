# Production Smoke Test & Deployment Checklist - AI Video Interviewer

**Version:** Post-Hardening (May 2026)
**Scope:** Full validation of authentication, WebRTC, AI interview flow, HR intervention, proctoring, reporting, Redis resilience, TURN refresh, and deployment integrity.

## Smoke Test Checklist

| Test Case ID | Description | Steps to Execute | Expected Result | Pass / Fail | Notes |
|--------------|-------------|------------------|----------------|-------------|-------|
| **AUTH-01** | HR login (Firebase email/password) | Navigate to `/login`, enter HR credentials (`hr@example.com` / valid password). | Redirect to HR dashboard, role recognized as `hr`, no redirect loops. | ☐ Pass / ☐ Fail | Use real Firebase user, not hardcoded. |
| **AUTH-02** | Candidate login via shared interview link | Open a unique candidate link (generated by HR). | Link loads, candidate not required to sign up; anonymous Firebase auth occurs silently. | ☐ Pass / ☐ Fail | Check browser console for Firebase anonymous UID. |
| **AUTH-03** | Expired candidate token (simulate 7 days later) | Manually modify `exp` claim in token or wait; attempt to open link. | Backend returns 401; frontend re-authenticates via Firebase anonymous and retries; link works again. | ☐ Pass / ☐ Fail | Verify log shows token expired and refresh. |
| **CAMPAIGN-01** | HR creates a new interview campaign | HR dashboard -> "New Campaign" -> enter title, description, skills, optional resume -> submit. | Campaign saved; 20-question pool generated (dataset + Gemini). | ☐ Pass / ☐ Fail | Check Firestore `campaigns` collection. |
| **CAMPAIGN-02** | HR generates an emailed candidate link | Campaign details -> "Send Email Link" -> enter candidate email -> send. | Email received (Resend) with unique link; link stored in Firestore `links`. | ☐ Pass / ☐ Fail | Use a test email inbox. |
| **CANDIDATE-01** | Candidate joins interview, uploads resume | Click link -> grant camera/mic -> fill basic info -> upload PDF/DOCX resume. | Resume parsed; match score calculated (skills, experience); question difficulty selected based on score. | ☐ Pass / ☐ Fail | Verify score >70 -> advanced questions appear. |
| **INTERVIEW-01** | AI speaks first question (TTS) | After resume upload, AI says "Question 1: ..." (English, Hindi, or Telugu based on setting). | Audio plays; dialogue history shows question; candidate sees timer. | ☐ Pass / ☐ Fail | Test all three languages via voice selector. |
| **INTERVIEW-02** | Candidate answers by voice | Click "Record" -> speak answer -> stop. | Audio transcribed (Google Speech) -> answer evaluated by Gemini -> score and feedback shown. | ☐ Pass / ☐ Fail | Check that evaluation appears within 5 seconds. |
| **INTERVIEW-03** | Candidate answers by code editor (if coding question) | Question type "code" -> Monaco editor appears -> type code -> submit. | Code evaluated (no copy-paste allowed); feedback on correctness, style. | ☐ Pass / ☐ Fail | Try Ctrl+C / Ctrl+V - should be blocked. |
| **PROCTORING-01** | Tab-switch detection | During interview, switch browser tab or open another app. | Warning modal appears; beep sound plays; violation logged. After 5 violations, interview disqualified. | ☐ Pass / ☐ Fail | Check `proctoring-alert` socket event. |
| **PROCTORING-02** | Face missing detection | Turn away from camera for 5 seconds. | Warning "Face not detected" appears; alert sent to HR observer. | ☐ Pass / ☐ Fail | face-api.js must load models. |
| **HR-OBSERVE-01** | HR joins observer room | HR opens `/observe/:campaignId` while candidate is live. | HR sees candidate's video stream, current question, answer being typed, proctoring alerts. | ☐ Pass / ☐ Fail | WebRTC connection must use TURN; check `chrome://webrtc-internals` for `relay`. |
| **HR-SPEAK-01** | HR requests to speak, candidate accepts | HR clicks "Request to Speak" -> candidate sees popup -> "Accept". | HR's camera/mic stream appears on candidate side; AI pauses; two-way audio works. | ☐ Pass / ☐ Fail | Verify both sides hear each other. |
| **HR-SPEAK-02** | HR ends speaking, AI resumes | HR clicks "End & Resume AI". | HR's media tracks stop; AI continues with next question. | ☐ Pass / ☐ Fail | No echo or lingering tracks. |
| **REPORT-01** | Final evaluation generation | Candidate completes all 5 questions. | Backend generates report with overall score (0-100), recommendation (Hire/Maybe/Reject), strengths/weaknesses. | ☐ Pass / ☐ Fail | Report stored in Firestore `sessions` collection. |
| **REPORT-02** | HR views and exports PDF | HR opens campaign -> sessions -> click report -> "Export PDF". | PDF downloads with score breakdown, answers, feedback, proctoring summary. | ☐ Pass / ☐ Fail | Use jsPDF; check formatting. |
| **REDIS-01** | Simulate Redis failure (production mode) | Stop Redis container; attempt to join a new room (candidate link). | Signaling server health endpoint (`/health`) returns 503; candidate connection fails gracefully with error message. Existing rooms may be lost - acceptable fallback. | ☐ Pass / ☐ Fail | Set `NODE_ENV=production` and `REDIS_URL` invalid; verify server rejects. |
| **REDIS-02** | Redis recovery after failure | Restart Redis; join a new room. | Signaling server reconnects, rooms are stored in Redis again. Health returns 200. | ☐ Pass / ☐ Fail | |
| **TURN-01** | TURN credential refresh | After 60 minutes of ongoing call, check `webrtcConfig.js` watch function. | TURN credentials are re-fetched; peer connection does not break (ICE restart triggered if needed). | ☐ Pass / ☐ Fail | Monitor console for `[TURN] Refreshing credentials`. |
| **TURN-02** | ICE failure recovery | Simulate network change (e.g., switch VPN). | `pc.oniceconnectionstatechange` -> `failed` -> `refreshTurnCredentials()` -> new offer/answer cycle; video resumes. | ☐ Pass / ☐ Fail | May take up to 20 seconds. |

```bash

git reset --hard <commit-hash>
git push --force origin main
```

Then re-deploy.
